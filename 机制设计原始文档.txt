### 核心架构：记忆引擎
* **上下文记忆：** 包含状态区和记忆区，状态区包含角色人设提示词【设定好后对话中基本固定不变】、用户信息【随对话更新】、角色情绪状态【随对话更新】，记忆区仅加载当前活跃权重大于某个阈值的前10个事件的索引文本、随机扰动加入的1个灵光一闪事件的索引文本以及最近 15 轮（滑动窗口）的原始对话。
* **短期巩固：** 当窗口滑动满足阈值（如对话满 10 轮），后台静默异步触发 LLM 任务，将近期对话打包为“记忆片段”并挂靠至相关索引事件/若无相关索引事件，则新建一个，同时提取可能存在的用户信息。此次请求不会进入上下文，但是请求后获得的记忆片段中的情绪状态emotion字段和提取到的用户信息会更新到上下文的状态区。引入版本锁避免脏读，在上一个异步巩固任务未结束时不允许触发下一个。
* **自动检索：当用户发送消息时，App 在本地用极轻量的关键字分词算法，静默去 SQLite 匹配最相近的 3 条记忆事件，将其活跃权重刷新，这样它就会被直接拼装到上下文的状态区，LLM 全程只负责充满情感地回复。
* **做梦：** 离线后台任务。根据活跃权重（时间衰减 + 调用频次）冻结冷数据，并合并可合并的同类型记忆事件。

### 记忆流转生命周期
* **步骤一：产生记忆。** 用户与 AI 交流。
* **步骤二：提取入库。** 每 10 轮对话，后台 LLM 提取记忆片段，包括 `[所属事件,概要, 情绪]`和用户信息。系统根据指令更新数据库。
* **步骤三：日常闲聊。** 上下文中自然注入高优活跃事件文本，AI 可以与用户无缝衔接话题。
* **步骤四：检索信息。** 用户提问“去年夏天的海岛游”。查询时间在去年，关键词分词为夏天、海岛的记忆事件注入提示词。
* **步骤五：做梦。** 当事件数量超过一个阈值或经过一段较长时间，系统拉取过去的记忆事件，LLM 将琐碎记忆事件（如多个探讨买车的事件）折叠为一个记忆事件（`[3月一直在为买车纠结]`），并遗忘不重要的细枝末节。

### 关键数据结构定义

**表一：记忆片段**

| 字段名 | 类型 | 说明 | 示例 |
| --- | --- | --- | --- |
| `index` | Int | 所属的父类索引记忆事件ID | `[17]` |
| `timestamp` | DateTime | 事件发生的时间（系统自动获取） | `2026-07-07 20:00:00` |
| `summary` | String | 50字以内的核心概要 | `[user]决定放弃去北海道，改去冲绳潜水，因怕冷。` |
| `emotion` | String | 此刻的情绪、状态 | ``[user]现在有些期待、略带纠结，你想帮帮他，但不知道怎么做` |

**表二：记忆事件**

| 字段名 | 类型 | 说明 | 示例 |
| --- | --- | --- | --- |
| `index` | Int | 索引记忆事件ID | `[17]` |
| `event_text` | String | 记忆事件的索引文本 | `[[user] [time]想去日本旅行，并与你探讨了旅行规划]` |
| `timestamp` | DateTime | 事件发生的时间（系统自动获取） | `2026-07-07 20:00:00` |
| `active_weight` | Int | 活跃度权重 (1 - 100，系统自动调整，根据最后一次被调用的时间自然衰减，例如刚生成新记忆片段或刚被刷新的事件活跃权重是100，随时间按照艾宾浩斯遗忘曲线递减，采用惰性求值 降低性能开销。)  | `85` |
| `last_accessed` | DateTime | 最后一次被调用的时间| `2026-07-07 20:01:00` |

*变量描述[user]是用户名，注入时会显示为用户的昵称，[time]是记忆事件的相对时间，例如系统获取的当前时间是9月，事件发生的时间是7月，最后一次调用的时间是8月，那这个time注入的时候就会显示“前两个月”。以上转换发生在从数据库取出索引、准备注入上下文的拼装阶段，存储时以[user]、 [time]存储。

为了快速跑通最小可行性产品 (MVP) 并支持后续的高频调优，在架构初始化阶段，必须将以下四类参数解耦并设置为可快捷自定义的配置项：

### 1. 提示词与模板层 (Prompt & Templates)

这是调优最频繁的重灾区。需要根据测试中出现的幻觉或遗漏，不断重写指令。

* **`system_prompt`**: 角色人设提示词。
* **`extraction_prompt`**: 后台记忆与用户信息提取的系统指令。必须包含严格的 JSON Schema 定义和冲突处理原则。
* **`state_injection_template`**: 状态区渲染模板。即后台将数据库查询到的 JSON 组合成前端 Markdown 注入上下文的占位符模板（如 `基础信息：{{nickname}}，所在地{{location}}`）。
* **`dream_consolidation_prompt`**: 离线做梦（长期记忆折叠）的指令。用于控制 AI 做梦的步骤， 触发同一主题记忆折叠的事件条数阈值以及保留更多客观事实还是主观情绪的决策。

### 2. 记忆流转阈值 (Flow Thresholds)

这些参数直接决定了系统的计算成本（Token 消耗）和响应延迟。

* **`consolidation_window_turns`**: 触发短期巩固的对话轮数（暂定为 10）。
* **`context_active_events_limit`**: 每次对话上下文中加载的高权重记忆事件数量上限（暂定 10 条）。

### 3. 权重与衰减策略 (Weight & Decay)

控制 AI 的“遗忘速率”与“意外感”。
* **`ebbinghaus_decay_rate`**: 时间衰减系数。决定事件的活跃权重随时间流逝下降的艾宾浩斯系数。
* **`epiphany_trigger_probability`**: “灵光一闪”随机事件的触发概率（例如 0.05）。

### 4. 模型路由与推理参数 (Model Routing)

后台结构化任务与前台发散性聊天的模型要求完全不同。

* **`background_extraction_config`**: 指向后台提取任务的模型配置。建议配置低成本模型，且 `temperature` 必须锁定为 0（或 0.1），开启 JSON Mode，追求结构绝对稳定。上下文轮数需要与consolidation_window_turns一致。
* **`foreground_chat_config`**: 指向用户交互的前台模型配置。上下文轮数`temperature` 建议暴露在 0.6 - 0.8 之间，以保证回复的同理心、幽默感和拟人化张力。
