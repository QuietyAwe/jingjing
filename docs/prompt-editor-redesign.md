# 提示词编辑器重构需求文档

## 一、现状分析

### 1.1 当前架构（消息发送顺序）

```
┌─────────────────────────────────────────────────────────┐
│ API 请求 messages 数组                                    │
├─────────────────────────────────────────────────────────┤
│ [0] system  → 系统人设          ← 稳定，高缓存命中        │
│ [1] system  → 状态区(用户信息)  ← 巩固窗口内稳定          │
│ [2..N]      → 对话历史          ← 前缀相对稳定            │
│ [N+1] system → 记忆区(事件)     ← 每次检索可能变化        │
│ [N+2] user  → 用户最新消息      ← 每次新的                │
└─────────────────────────────────────────────────────────┘
```

### 1.2 当前设置页字段

| 字段 | 说明 | 问题 |
|------|------|------|
| 系统人设 | system_prompt | ✓ 位置合理 |
| 上下文提示词模板 | context_template | ✗ 已废弃，仍显示，用户困惑 |
| 状态区注入模板 | state_injection_template | ✓ 但名称不直观 |
| 记忆区注入模板 | memory_injection_template | ✓ 但名称不直观 |
| 记忆事件格式 | memory_event_template | ✓ |
| 提取指令 | extraction_prompt | 与聊天流程无关，混在一起 |
| 做梦折叠指令 | dream_consolidation_prompt | 与聊天流程无关，混在一起 |
| 冷启动模板 | cold_start_template | 特殊场景，混在一起 |

### 1.3 核心问题

1. **废弃字段仍在**：`context_template` 已不使用，但用户仍能看到并编辑
2. **分组无逻辑**：8个字段平铺，无层次关系
3. **架构不直观**：用户无法理解"系统人设"和"状态区"是分开发送的
4. **缓存优化无感知**：用户不知道哪个稳定、哪个易变
5. **混合了不同场景**：聊天模板、后台任务模板、特殊场景模板混在一起

## 二、重构方案

### 2.1 信息架构（三级分组）

```
提示词设置
├── 💬 聊天提示词（按发送顺序排列）
│   ├── ① 系统人设         → [system] message（稳定，缓存命中）
│   ├── ② 状态区注入模板   → [system] message（巩固窗口内稳定）
│   └── ③ 记忆区注入模板   → [system] message（每次检索可能变化）
│       └── └── 记忆事件格式 → 单条事件的渲染模板
│
├── 🔧 后台任务提示词
│   ├── 记忆提取指令       → 巩固流使用
│   └── 做梦折叠指令       → 做梦流使用
│
└── 🚀 特殊场景
    └── 冷启动模板         → 空库首次对话
```

### 2.2 各编辑区设计

#### ① 系统人设

- **位置**：第一项，最稳定
- **内容**：角色定义、人格设定
- **可用变量**：`[user]`、`[now]`
- **说明文案**：「定义 AI 的角色与人格。此部分作为独立消息发送，缓存命中率最高。」

#### ② 状态区注入模板

- **位置**：第二项，较稳定
- **内容**：用户信息的展示格式
- **可用变量**：`[user]`、`[now]` + 14个状态变量
- **说明文案**：「定义用户信息的注入格式。此部分在10轮对话内保持稳定，空字段自动隐藏。」
- **预览**：显示当前用户数据的渲染结果

#### ③ 记忆区注入模板

- **位置**：第三项，易变
- **内容**：记忆区的整体结构
- **可用变量**：`{{event_list}}`、`{{epiphany}}`
- **说明文案**：「定义记忆区的整体结构。每次检索可能注入不同事件。」
- **子编辑**：嵌套「记忆事件格式」编辑

#### ④ 记忆事件格式（嵌套在记忆区下）

- **位置**：记忆区注入模板下方，缩进显示
- **内容**：单条记忆事件的格式
- **可用变量**：`[user]`、`[now]`、`[time]`、`{{event_text}}`
- **说明文案**：「定义单条记忆事件的格式。[time] 为相对于事件发生时间的描述。」

#### ⑤ 记忆提取指令

- **位置**：后台任务分组下
- **内容**：巩固流的 LLM 提取指令
- **可用变量**：无（纯指令文本）
- **说明文案**：「每10轮对话触发一次，从对话中提取用户信息和记忆片段。」

#### ⑥ 做梦折叠指令

- **位置**：后台任务分组下
- **内容**：冷数据语义折叠指令
- **可用变量**：无（纯指令文本）
- **说明文案**：「闲置时自动触发，将琐碎记忆事件折叠为概括性事件。」

#### ⑦ 冷启动模板

- **位置**：特殊场景分组下
- **内容**：空库首次对话的引导人设
- **可用变量**：`[user]`
- **说明文案**：「当数据库为空时，与系统人设组合使用，引导用户自我介绍。」

### 2.3 删除项

- **`context_template`**：已废弃，从设置页移除（配置保留，不影响旧数据）

## 四、变更影响

| 影响项 | 说明 |
|--------|------|
| 删除字段 | `context_template` 从 UI 移除，配置数据保留 |
| 新增分组 | 三个分组：聊天提示词、后台任务、特殊场景 |
| 排序调整 | 按消息发送顺序排列 |
| 说明文案 | 每个字段新增说明，解释其作用和缓存特性 |
| 变量选择器 | 已有功能，按新分组调整可用变量 |

---

## 五、不变更项

- 各模板的实际渲染逻辑（assembler.ts）
- 变量替换规则
- 默认配置值（defaultConfig.json）
- 后台任务的执行逻辑
